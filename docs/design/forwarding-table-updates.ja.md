## 先行 3 モデルの要約

**sdplane — ユーザ空間 RCU 下の RIB ポインタスワップ。** run-to-completion
ワーカー(lcore ごとに 1 つ)が共有 RIB ポインタを QSBR RCU
(`rcu_dereference`、liburcu-qsbr)越しに読む。更新は生きているテーブルに
決して触れない: `rib-manager` が**完全な新 RIB** を構築し、ポインタを
アトミックに差し替え、グレースピリオドを待って旧 RIB を解放する。ワーカーは
停止・スピン・バリアのいずれも受けず、中間状態を観測することもない
(マイクロループ無し: 読み手はテーブル全体のスナップショットを見る)。
コストはコントロールプレーンに落ちる: 更新のたびに **O(N 総経路)** の
再構築とグレースピリオドが掛かり、メモリは一時的に RIB を 2 つ抱える。
論文はこれが現在の `rib-manager` の選択であって RCU に内在するものでは
ないと述べる — 差分更新に移行してもワーカー非干渉性は保てる。FIB 自体は
意図的に素朴な 2 ビットストライドのトライである(v4 で ≤16 段、v6 で
≤64 段)。RCU が読み手の安全性を構造から切り離すため、より高速な構造
(radix、poptrie、ハッシュ)を並行制御を一切設計せずに差し替えられる。

**VPP IPv4 — `ip4_mtrie`(16-8-8)、構造に焼き込まれた並行制御。**
リーフは 32 ビットワード(終端 = ロードバランスインデックス、さもなくば
次 ply のプールインデックス)。ルックアップは依存配列参照 ≤3 回。各スロットは
そのアドレス範囲を覆う最も特定的な経路を保持し、LPM 意味論はスロット単位で
成立する。更新: リーフ書き換えはアトミックな release store。新しい部分木は
**完全に構築してから**親リーフへの 1 回のアトミックストアで**公開**する。
削除はスロットをカバー経路へアトミックに復帰させ、ply をフリーリストへ返す。
唯一の穴: ply はグローバルプール(`ip4_ply_pool`)に住んでおり、**プール
拡張がノード配列全体を realloc・移動させ得る** — 木を下っている最中の
読み手が旧アドレスに取り残される — ため、VPP はこの場合だけ
`vlib_worker_thread_barrier_sync` で*全*ワーカーを停止させる。発生は
限定的(過去最大の高水位を超えて成長する間のみ。解放された ply は再利用
され、プールは縮小も移動もしない)だが、成長中のフルフィード下では断続的に
発生し、Rx リングの損失/レイテンシとして現れる。

**VPP IPv6 — `bihash`、バケット単位の楽観的並行制御。** LPM は木の走査では
ない: テーブルに実在するプレフィックス長を長い順に、マスクしたキーの
ハッシュを 1 長あたり 1 回引く(1 プローブ = CRC32C 3 連鎖)。更新は単一の
バケットロックビットを立て、そのバケットを別メモリで変異・分割し、64 ビットの
バケット記述子をアトミックに差し替える。読み手はロックフリーで、seqlock 流の
世代再検証を行う(衝突した読み手は短くスピンするか 1 回リトライする)。
**決してバリアしない** — ただしルックアップコストは実在プレフィックス長の
数 P に比例し(IPv6 DFZ で約 40)、どの経路にも一致しない宛先は全長を試す。

「cradle 現状」= 実装済みの `FIB4`/`FIB6` LPM トライ + ハッシュマップ。

## cradle はスペクトラムのどこに位置するか

論文の中心的な観察は、sdplane と VPP の違いが*並行制御がどこに住むか*に
あるということである: **転送構造の内部**(VPP — 公開規律、バケットロック、
そして逃げ道としてのバリア)か、**独立した機構への分離**(sdplane — RCU が
読み手の安全性を*構造*ではなく*ポインタ*の性質にし、アルゴリズム差し替えの
自由を買う)か。

cradle は正真正銘の第三の位置にある: **カーネルへ委譲された並行制御**で
ある。すべての eBPF マップ型は検証済みの読み手/書き手意味論を備えて出荷
される(BPF プログラムは常に RCU 下で走り、要素更新は要素粒度でアトミック)。
したがってデータパスも制御プレーンも並行制御を一切実装しない。対価は
sdplane の自由の逆である: 構造のメニューはカーネルが固定する — マップ型を
*合成*するまでは。それこそが DIR-24-8 のやっていることである(`Array`
マップ上のインデックス演算。cradle が自前で供給すべき唯一の並行制御規則 —
fill-then-flip の順序付け — はデータパスではなくユーザ空間の展開エンジンが
強制する)。

論文の知見と cradle の設計との間の 3 つの共鳴:

1. **DIR-24-8 の fill-then-flip は mtrie の complete-then-publish である。**
   [`large-fib.md`](large-fib.md) は VPP の部分木規律に独立に到達した —
   `TBL8` グループを完全に埋めてから `TBL24` ワードをアトミックにそこへ
   フリップし、リサイクル前にフリップで外す。同じ不変条件(「読み手は旧
   カバーか完成した新ノードを見る。作りかけは決して見ない」)を、C 構造体の
   内部ではなくマップ ABI のレベルで表現したものである。
2. **完全事前割り当てが mtrie 唯一の stop-the-world ケースを消す。** VPP が
   バリアするのはノードプールが成長時に*再配置*されるからである。eBPF の
   `Array` マップは決して再配置されず、カーネル LPM トライはノード単位の
   ポインタ連結である — 全ワーカー停止のケースは cradle のモデルには構造的に
   存在し得ない。トレードはロード時に固定される容量(`--fib4-mode`
   サイジング)、すなわち cradle は*成長を拒否することで* VPP のルックアップ
   形状をその成長バリア抜きで買っている。
3. **sdplane のテーブル全体スナップショットは、他の誰も持たない唯一の
   性質である。** 構成上マイクロループゼロ。cradle の一貫性は順序付け規律
   付きの要素/ワード単位であり — VPP ともあらゆるハードウェア FIB とも同じ
   結果整合性の窓である。スナップショット意味論が欲しくなった場合の eBPF
   イディオムは存在する: **map-in-map**(FIB を保持する `ARRAY_OF_MAPS` の
   外側マップ)であり、外側エントリのフリップはまさに sdplane の RIB
   ポインタスワップである — そして sdplane のトレードもそのまま輸入する:
   スワップごとの O(N) 再構築、両世代が生きる間のメモリ倍増。
   [`large-fib.md`](large-fib.md) にオプションとして記録した。提案しないのは
   DFZ 規模の経路チャーンが O(Δ) を求めるからである。

チャートを読む際に念頭に置くべき 2 つの正直な非対称性:

- **読み手の配置。** sdplane と VPP は決定的レイテンシのために専用ポーリング
  コアを燃やす。cradle の読み手は NAPI が着地した CPU のカーネル softirq で
  走る — 予約コア無し、ビジーポール税無し、ただしレイテンシ隔離も無し。
  この違いは更新機構とは直交するが、「ワーカーを決して止めない」という
  すべてのセルに色を付ける: cradle には止める*べき*ワーカーがそもそも
  存在しない。
- **ライタのボトルネックは消えたのではなく移動した。** sdplane は O(N)
  再構築を払い、VPP は時折のバリアを払い、cradle は現状トライ単位のライタ
  スピンロックを払う(カーネルが `LPM_TRIE` 更新をシリアライズする)。
  それこそが [`large-fib.md`](large-fib.md) の存在理由である収束時間の
  天井であり — DIR-24-8 のライタパスはロックフリーなワードストアで、
  ライタ側の唯一の上限としてバッチ syscall スループットが残る。
