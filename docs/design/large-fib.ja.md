# cradle-rs 大規模 FIB(DIR-24-8)— 設計

> eBPF データプレーンに 100 万の学習経路を: IPv4 の LPM トライを DPDK 流の
> DIR-24-8 直接インデックス FIB に置き換える — O(1)〜O(2) のフラットな配列
> ルックアップ、パケットごとのロック無し、フル DFZ 容量 — しかも `FibHandle`
> の継ぎ目は不変のまま。

ステータス: **フェーズ 1〜2 実装済み。** フェーズ 1: シャドウ + 展開
エンジン(`crates/cradle/src/dir24.rs`、参照 LPM に対するプロパティテスト
付き)、`TBL24`/`TBL8`/`DEFAULT4` のパック化ワードデータパス、ロード時
サイジング付き `--fib4-mode` / `"fib4_mode"`、3 つのカウンタ、
`cradle_dir24` BDD スモーク。フェーズ 2: バルク初期ロードパス
(`route_add_bulk` — バッチごとに影響 /24 ブロックを 1 回だけ同期)、
`AddRoute4Batch` RPC(設定ウォーカーが使用)、`GetFibSummary` /
`cradle ctl fib`、DFZ 形状の `ctl gen-routes` ジェネレータ、そして
`cradle_bigfib` BDD(**100 万経路の FIB** をロードし、3 つのルックアップ
パスすべてで転送し、負荷中の撤回まで検証。実測: **100 万経路をリリース
ビルドで 2.0 秒 / デバッグで 9.5 秒、約 49 万経路/秒**、要素単位のマップ
書き込みのまま — `BPF_MAP_UPDATE_BATCH` の代替案はこの規模では不要と実測
された)。LPM がデフォルトモードのまま。フェーズ 2 の残り:
`BPF_PROG_TEST_RUN` ルックアップレイテンシハーネス(後続 PR)。以下の
フェーズ 3〜4 は設計のままである。[`architecture.md`](architecture.md) の L3 パスの
上に構築される。オーバーレイ設計([`mpls.md`](mpls.md)、[`srv6.md`](srv6.md)、
[`evpn-vxlan.md`](evpn-vxlan.md))はいずれも本設計に依存しないが、三者が収束
する共有の per-VRF FIB は、グローバルテーブルの行き先がここであることを
踏まえて構築されるべきである。

(英語原文: [`large-fib.md`](large-fib.md))

## 目的とスコープ

cradle の差別化点は、**学習した**経路が eBPF FIB をプログラムすることである —
そして「学習した」の最も過酷な形は IPv4 のフル DFZ フィード、今日およそ 100 万
プレフィックスである。現在の FIB はそれを有用な形では保持できない:

```rust
// crates/cradle-ebpf/src/main.rs:50
static FIB4: LpmTrie<[u8; 4], FibEntry> = LpmTrie::with_max_entries(4096, 0);
```

`max_entries` を引き上げることは解決策ではない。`BPF_MAP_TYPE_LPM_TRIE` は
100 万エントリを問題なく**保持**する(`BPF_F_NO_PREALLOC` が強制される。トライ
ノード約 200 万個 ≒ 150〜250 MB)— ただしその規模ではルータであることを
やめてしまう。理由は 2 つあり、いずれも eBPF ではなくこのマップ型の性質である:

1. **パケットごとのルックアップコスト。** カーネルの LPM トライはパス圧縮
   された二分トライであり、最長一致の下降は IPv4 で最大 **32 ノード訪問**
   (IPv6 で 128)、各訪問はポインタ追跡でありキャッシュミスになりやすい。
   フルテーブルに対してワイヤレートで叩けば、この walk が pps の上限
   そのものになる。
2. **更新のシリアライズ。** トライ全体を 1 つのスピンロックが保護する。
   100 万経路にわたる BGP 収束ストームは全 insert/withdraw をシリアライズ
   する — コントロールプレーン側(収束時間)のコストだが、DFZ のチャーンでは
   現実に効く。

解決策は、ハードウェアに近いソフトウェアルータが使う構造 — **DIR-24-8**
(DPDK `rte_lpm`。VPP の `ip4-mtrie` は 16-8-8 の親戚)である。メモリを
決定性と引き換えにする — そして eBPF では `Array` マップとインデックス演算
だけで表現できる。

較正のために: これは意図的に Cilium が保持する範囲を超えている。Cilium の
L3 マップ(`cilium_ipcache`)はデフォルト 512k エントリで、ポリシーと
トンネル選択のためのアイデンティティ/CIDR キャッシュであり、ネクストホップ
FIB ではない。フル DFZ の eBPF FIB は、cradle のテーゼを具体化したものである。

## 1 ページで分かる DIR-24-8

フラットな配列が 2 つ:

- **`TBL24`** — 宛先アドレスの上位 24 ビットでインデックスする:
  2²⁴ = 16,777,216 スロット。各 4 バイトエントリ ⇒ **64 MiB、事前割り当て**。
  スロットはパケットを直接解決する(その /24 を覆う経路がすべて /24 以下の
  場合)か、tbl8 グループを指す。
- **`TBL8`** — 256 エントリのグループのプール。より長いプレフィックス
  (/25〜/32)を含む /24 は 1 グループを得て、アドレスの下位 8 ビットが
  その中をインデックスする。

ルックアップは、DFZ 型トラフィックの 99% 以上(プレフィックス ≤ /24)で
配列アクセス 1 回、最悪でもちょうど 2 回。ループ無し、配列参照以外の
ポインタ追跡無し、ロック無し — `Array` マップの読み出しは RCU 保護の
プレーンなロードである。コストは**更新時**に移る: /24 より短い経路は
*展開*される — /16 は `TBL24` スロット 256 個を書く — が、読み出し優位の
DFZ にとってこれは正しいトレードである。

なぜ VPP の 16-8-8 ではなく 24-8 か: 16-8-8 はルートを 256 KiB に縮め、
/16 を 1 回の書き込みにするが、一般的なルックアップは最大 3 回の依存
ロードになり、一般的な*更新*(DFZ の最頻プレフィックスである /24)も
リーフに触れる。DFZ を欲しがるクラスのマシンにとって 64 MiB のルートは
安い。我々は可能な限り平坦な読み出しパスを買う。メモリフットプリントが
問題になる場合の代替は 16-8-8 であり(そして v6 の形は恐らくそれになる —
後述)。

## マップコントラクト(`cradle-common`)

### パック化エントリ

`FibEntry{nexthop_id: u32, flags: u32}` は 8 バイトである。直接インデックス
テーブルは同じ情報を **4 バイトワード**にパックして `TBL24` を 64 MiB に
保つ:

```rust
/// Packed DIR-24-8 slot. Layout (bit 31 .. bit 0):
///   [31]     FIBW_VALID
///   [30]     FIBW_TBL8    — low bits are a TBL8 group index, not a nexthop
///   [29:26]  flags        — FIB_F_BLACKHOLE | LOCAL | CONNECTED | ECMP (4 bits)
///   [25:0]   nexthop_id (or group index when FIBW_TBL8): 64M ids
pub type FibWord = u32;

pub const FIBW_VALID: u32 = 1 << 31;
pub const FIBW_TBL8:  u32 = 1 << 30;
pub const FIBW_FLAGS_SHIFT: u32 = 26;
pub const FIBW_ID_MASK: u32 = (1 << 26) - 1;
```

既存の `FIB_F_*` ビットは 4 ビットフィールドに余裕を持って収まる(今日は
`1<<0..1<<3`)。`nexthop_id` は 26 ビットを保ち、いかなるネクストホップ/
グループ数をもはるかに超える。アンパックされた `FibEntry` は LPM パスと
gRPC 面のためにそのまま残る — パック化は DIR テーブルの内部事情である。

### マップ

```rust
#[map] static TBL24: Array<FibWord> = Array::with_max_entries(1 << 24, 0);
#[map] static TBL8:  Array<FibWord> = Array::with_max_entries(TBL8_GROUPS * 256, 0);
```

- `TBL8_GROUPS` のデフォルトは **4096**(4 MiB): /24 より長いプレフィックスを
  含む /24 あたり 1 グループ。v4 の DFZ は /24 超をほとんど伝搬しないため、
  グループを消費するのは*ローカル*な状態 — ホスト経路、VIP ローカル、
  connected の /31〜/32 — である。4096 は十分に余裕があり、ロード時の
  ノブでもある(後述)。
- どちらもプレーンな(共有・事前割り当ての)`Array` マップ: 読み出し優位、
  ロックフリーな読み手、要素単位のアトミックな 4 バイト更新。

### コンパイル時分岐ではなくロード時サイジング

aya の `EbpfLoader::set_max_entries` はマップ生成前にサイズを変更できる。
`cradle --fib4-mode {lpm,dir24}`(JSON 設定にもある)がエンジンを選ぶ:

- `lpm`(小規模デプロイのデフォルト): `TBL24`/`TBL8` は 1 エントリで
  生成される — 64 MiB のコストは端的に存在しない。`FIB4` は今日のまま
  動き続ける(その `max_entries` もノブになる)。
- `dir24`: フルサイズの配列。`FIB4` は 1 エントリに縮む。

コンパイル済みオブジェクトは 1 つ、データパスも 1 つ。データパスは設定
フラグからパケットごとにエンジンを選ぶ(すでに読んでいる `Array<u32>` の
設定ワード 1 個は安い。1 エントリ配列でのミスはどのみち他方のエンジンに
フォールスルーするので、このフラグは正しさの要件ではなくファストパスの
ヒントである)。

## データプレーンのルックアップ(`cradle-ebpf`)

`l3_forward_v4` の `FIB4.get(Key::new(32, dst))`(`main.rs:703`)を置き換える。
下流のすべて — `FIB_F_*` の処理、ECMP メンバー選択、TTL、
`bpf_redirect_neigh` — は不変である:

```rust
let idx24 = (u32::from_be_bytes(dst) >> 8) as u32;
let mut w = *TBL24.get(idx24).ok_or(())?;
if w & FIBW_TBL8 != 0 {
    let group = w & FIBW_ID_MASK;
    w = *TBL8.get(group * 256 + dst[3] as u32).ok_or(())?;
}
if w & FIBW_VALID == 0 {
    return Ok(TC_ACT_PIPE as i32);            // no route → host stack
}
let flags = (w >> FIBW_FLAGS_SHIFT) & 0xf;
let nexthop_id = w & FIBW_ID_MASK;
// → existing blackhole/local/ECMP/nexthop path
```

有界な配列参照 2 回、ベリファイアの問題は皆無。これは置き換える LPM 呼び出し
より*少ない*プログラム複雑度である。

### デフォルトルートはテーブルの外に置く

`0.0.0.0/0` を展開すると、デフォルトルートが変わるたびに `TBL24` の全
1670 万スロットを書くことになる。代わりに、invalid ワードのフォールスルーが
パントの前に 1 エントリの `DEFAULT4: Array<FibWord>` を調べる。これは
きわめて短いプレフィックスにも安価な逃げ道を与える(オペレータのテーブルが
病的な場合)— が、通常の /8(DFZ には一握り存在する)は普通に展開される:
65,536 のバッチ書き込みはミリ秒である。

## ユーザ空間側: シャドウトライ + 展開エンジン

今日 `Dataplane::route4_add` は LPM マップを直接書いており、**ユーザ空間の
経路シャドウは存在しない**(`crates/cradle/src/dataplane.rs`)。DIR-24-8 は
シャドウを必須にする: 展開は、影響を受ける各スロットについて、どの
プレフィックスが*最も特定的なカバー*かを知らねばならず — その情報は経路
集合全体だけが与える。

`cradle` クレートに新設:

- **シャドウトライ** — `(prefix → FibEntry)` のユーザ空間 radix トライ。
  プログラムされる*べき*ものの正である。(DPDK がテーブルエントリに焼き込む
  depth 情報の自然な置き場所でもある。depth をシャドウに置くことでカーネル
  側のワードは 4 バイトに保たれる。)
- **展開エンジン** — 1 つの経路 add/del をスロット書き込みに変換する:
  - `len ≤ 24`: カバーされる `2^(24-len)` 個の `TBL24` スロットそれぞれに
    ついて、このプレフィックスがスロットの現在のカバーより特定的である
    場合に**のみ**パック化ワードを書く(シャドウ参照)。削除は範囲を
    次善のカバー、または invalid に書き戻す。
  - `len > 24`: カバーする /24 に `TBL8` グループがあることを保証する
    (フリーリストから割り当て、シャドウから**完全に埋める** — ≤ /24 の
    カバーが背景スロットを埋め、より長いプレフィックスが自分のスロットを
    上書きする — *その後で* `TBL24` スロットをそこへ向けてフリップする)。
    削除はグループを直接エントリに畳み込むことがある。グループは**遅延して**
    (猶予期間を置いて)フリーリストに戻る。これにより、古い `TBL24` ワードを
    読んだ通過中のパケットが、別の /24 のために書き換え中のグループを
    インデックスすることは決してない。
  - すべての範囲書き込みは `BPF_MAP_UPDATE_BATCH`(カーネル ≥ 5.6、aya の
    バッチ操作)を通す: 100 万経路のフル初期ロードは約 150〜200 万スロット
    書き込みに展開され、数秒台前半でバルクロードされる。/16 のフラップは
    256 スロットのバッチ呼び出し 1 回である。

順序付けにより、いかなるロックも無しに読み手は一貫したビューを得る:
グループは `TBL24` が指す前に完成しており、`TBL24` はグループがリサイクル
される前に他所を指す。個々の 4 バイト更新はアトミックである。複数スロットの
展開の最中、テーブルは一時的に新旧のカバーを混在させる — これはあらゆる
ハードウェア FIB がチャーン中に持つのと同じ結果整合性の窓であり、LPM トライの
プレフィックス単位アトミック性も実際にはエンドツーエンドでこれ以上を約束して
いない(経路変更はすでに複数 RPC である)。

直観のための更新コストプロファイル:

| 変更 | スロット書き込み |
|---|---|
| /25〜/32 追加 | ≤ 256(グループ充填)+ フリップ 1。グループ既存なら 1 |
| /24 追加(DFZ 最頻ケース) | 1 |
| /20 追加 | 16 |
| /16 追加 | 256 |
| /8 追加 | 65,536(バッチ) |
| デフォルトルート | 1(`DEFAULT4`) |
| 100 万経路のフルフィード | 約 150〜200 万、バッチ |

そして決定的に: **ネクストホップのチャーンはスロット書き込みゼロである。**
cradle の経路はすでに `nexthop_id` を介して間接参照している(`NEXTHOPS`、
`NHGROUP`)— 80 万経路を新しいネクストホップへ動かすピアのフラップは、
*ネクストホップエントリまたはグループメンバー 1 つ*を書き換えるだけで、
巨大配列には決して触れない。ECMP をきれいにした間接参照こそが、DIR-24-8 を
保守可能にするものである。

## スナップショット意味論 — 検討し、見送り

スロット単位の更新は、複数スロットの展開が一時的に新旧のカバーを混在させる
ことを意味する — 標準的な結果整合性の窓である。sdplane 流の*テーブル全体
スナップショット*(マイクロループゼロ)の eBPF イディオムは存在する:
FIB を保持する `ARRAY_OF_MAPS` の外側マップで、外側エントリのフリップが
RIB ポインタスワップに相当する。これは対応するトレード — スワップごとの
O(N) 再構築と、両世代が生きている間のメモリ倍増 — も一緒に輸入する。DFZ の
チャーンレートでは間違ったトレードであり、本設計は O(Δ) のインプレース更新に
留まる。(sdplane / VPP / cradle の完全な比較は
[`forwarding-table-updates.md`](forwarding-table-updates.md) を参照。
DIR-24-8 の fill-then-flip グループ順序は VPP mtrie の complete-then-publish
規律であり、完全事前割り当てこそが mtrie のプール拡張バリアを cradle の
モデルから消すものである。)

## 変わらないもの

- **`proto/cradle.proto`** — `AddRoute4`/`DelRoute4`/`SetNexthop*` は
  手つかず。エンジンの交換は継ぎ目からは見えない。
- **zebra-rs のティー** — `CradleFib` は同じメッセージを送り続ける。DFZ
  フィードは単にその*量*が増えるだけである。(初期収束中はティーがバッチ
  フレンドリにバッファすべきだが、それはトランスポートのチューニングで
  あって API ではない。)
- **`FIB6`** — LPM トライのまま。v6 の DFZ は約 20〜25 万プレフィックスで、
  128 ビットキーは直接インデックスできない。20 万エントリのトライは許容
  できる。v6 のスケールが要求したときの答えは、`Array` マップ上の
  マルチビット・ストライドトライ(16-8-8-… 形)か、/48+/64 のハッシュ
  ファストパス + LPM フォールバックである — フェーズ 4、スケッチのみ。
- **オーバーレイ設計** — MPLS `POP_L3`、SRv6 `End.DT46`、EVPN Type-5 は
  いずれも「テーブルで IP ルックアップをする」。per-VRF テーブルは LPM
  トライがデフォルト(VRF は小さい)。フルフィードを運ぶ VRF は後日
  `ARRAY_OF_MAPS` で自前の DIR インスタンスを持てる。per-VRF の継ぎ目は
  *テーブルハンドル*を渡すべきであり、マップ型を仮定すべきではない —
  それが本書が他の設計にエクスポートする唯一の設計制約である。

## 可観測性

```
STAT_FIB4_TBL24_HIT   // resolved in one lookup
STAT_FIB4_TBL8_HIT    // two-lookup resolution
STAT_FIB4_DEFAULT     // fell through to DEFAULT4
```

加えて `cradle ctl fib summary`: エンジンモード、経路数、`TBL8` グループの
使用/空き、直近バルクロードの所要時間。BDD は ping だけから挙動を推測する
のではなく、これらをアサートする。

## テスト

正しさが先 — 展開エンジンは純粋なロジックであり、安いところで重いテストを
受ける:

- **プロパティテスト(ユーザ空間、eBPF 不要):** ランダムな経路集合と
  add/del 列を生成する。各ステップ後、サンプリングしたアドレスについて、
  DIR-24-8 テーブル(配列としてシミュレート)がシャドウトライ上の参照 LPM と
  同一に解決しなければならない。これが展開エンジンの正しさの議論の全体で
  あり、`cargo test -p cradle` で走る。
- **BDD `cradle_bigfib.feature`:** フォワーダ namespace が合成経路集合を
  ロードし(ジェネレータが DFZ 様の長さ分布で約 100 万プレフィックスを
  `ctl apply` 経由で放出)、その後: /24 経由の ping が成功する。/24 内の
  /28 経由の ping が tbl8 パスを取る(`fib4_tbl8_hit` が動く)。カバー
  されていたプレフィックスの撤回はカバー経路にフォールバックする。
  デフォルトルートが機能する。必須の `Scenario: Teardown topology` で
  終わる。
- **性能ハーネス(BDD ではない):** 1k / 100k / 1M 経路でテーブルを投入した
  TC プログラムへの `BPF_PROG_TEST_RUN`、LPM 対 DIR — 本設計を正当化する
  数字であり、CI のゲートには入れない。

## フェーズ分け

1. **フェーズ 1 — エンジン。** シャドウトライ + 展開エンジン + プロパティ
   テスト。`TBL24`/`TBL8`/`DEFAULT4` マップとパック化ワードのデータパス。
   ロード時サイジング付き `--fib4-mode`。カウンタ。LPM がデフォルトモードの
   まま。
2. **フェーズ 2 — スケール。** バッチ更新、遅延リサイクル付きグループ
   フリーリスト、バルク初期ロードパス、100 万経路 BDD + 性能ハーネスの
   数字。
